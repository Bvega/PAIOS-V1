Scenario
You are a junior backend developer at a new startup creating a movie review website. Your first major task is to build the core of the backend: a “Movie Finder” API. This API will not store any data itself; instead, it will act as an intermediary, fetching movie information from a public, external movie database. Your API will then provide cleaned-up, relevant data to the future front-end application.

This assessment will test your ability to structure a server, handle routes, interact with an external service, and manage configuration securely - all core skills for a backend developer.

Required Reading: RESTful Principles
Before you begin, it’s important to understand the principles of a RESTful API. REST (REpresentational State Transfer) is an architectural style that defines a set of constraints for creating web services. A well-designed RESTful API is predictable, easy to use, and scalable.

The key principles are:

Client-Server Architecture: The client (e.g., a front-end app) and the server (your Express app) are completely separate. The client only needs to know the URI (the endpoint) of the resource it wants to interact with.
Statelessness: Every request from a client to the server must contain all the information the server needs to fulfill it. The server does not store any information about the client’s state between requests. This makes the application more reliable and easier to scale.
Uniform Interface: This is the core of REST. It simplifies and decouples the architecture, which allows each part to evolve independently. It includes:
Resource-Based URLs: URLs should be intuitive and identify resources, not actions. Use nouns, not verbs.
Good: /api/movies/search
Bad: /api/getMovieSearchResults
HTTP Methods (Verbs): Use standard HTTP methods to perform actions on resources. For this assessment, you will only use GET, but a full CRUD application would also use:
GET: Retrieve a resource.
POST: Create a new resource.
PUT / PATCH: Update an existing resource.
DELETE: Remove a resource.
Standard Representations: Resources are decoupled from their representation. The client and server agree on a format, which is almost always JSON for modern web APIs.

### 1. Replace `src/controllers/movieController.js`

```js
// src/controllers/movieController.js
// ────────────────────────────────
// All business logic for the Movie Finder API.

import axios from 'axios';
import { OMDB_API_KEY } from '../config/index.js';

const OMDB_BASE_URL = 'http://www.omdbapi.com/';

/**
 * GET /api/search?title=<title>
 * Returns an array of movies matching the search term.
 */
export const searchMovies = async (req, res) => {
  const { title } = req.query;

  // 1) Guard-clause validation
  if (!title?.trim()) {
    return res
      .status(400)
      .json({ error: 'Title query parameter is required' });
  }

  try {
    // 2) Proxy call to OMDb
    const { data } = await axios.get(OMDB_BASE_URL, {
      params: { s: title.trim(), apikey: OMDB_API_KEY }
    });

    // 3) OMDb returns { Response: "False", Error: "Movie not found!" }
    if (data.Response === 'False') {
      return res.status(404).json({ error: data.Error });
    }

    // 4) Happy path
    return res.json(data.Search); // send the array only
  } catch (err) {
    console.error('OMDb search error:', err.message);
    return res
      .status(502)
      .json({ error: 'Upstream OMDb service unreachable' });
  }
};

/**
 * GET /api/movies/:id
 * Returns full details for a specific IMDb ID.
 */
export const getMovieDetails = async (req, res) => {
  const { id } = req.params;

  if (!id?.trim()) {
    return res.status(400).json({ error: 'ID path parameter is required' });
  }

  try {
    const { data } = await axios.get(OMDB_BASE_URL, {
      params: { i: id.trim(), plot: 'full', apikey: OMDB_API_KEY }
    });

    if (data.Response === 'False') {
      return res.status(404).json({ error: data.Error });
    }

    return res.json(data);
  } catch (err) {
    console.error('OMDb details error:', err.message);
    return res
      .status(502)
      .json({ error: 'Upstream OMDb service unreachable' });
  }
};
```

### What is a **repository secret** in GitHub Actions?
* A **secret** is an encrypted key-value pair stored by GitHub.  
* During a workflow run it is injected as an **environment variable** (never written to logs or persisted in artefacts).  
* Typical use-cases: API keys, database passwords, signing certs—anything you **must not commit** to the repo.

Because our tests hit the live OMDb API, the CI runner needs the same `OMDB_API_KEY` you have locally—but we must keep it private.  
That’s why the workflow references:

```yaml
env:
  OMDB_API_KEY: ${{ secrets.OMDB_API_KEY }}
```

When the job starts, GitHub replaces `${{ secrets.OMDB_API_KEY }}` with the value you store in **Settings → Secrets**.

---

## Step-by-step: add `OMDB_API_KEY` to your repo

1. **Open your repository on GitHub** in a browser.  
2. Click **Settings** (top navigation bar).  
3. In the left sidebar choose **Secrets and variables → Actions**.  
4. Press **New repository secret**.  
5. **Name** → `OMDB_API_KEY` *(must match the name in the workflow)*  
6. **Value** → paste your actual OMDb key, e.g. `k3y1abc2`.  
7. Click **Add secret**.  
   > You’ll now see it listed, but the value is permanently hidden—only “Last updated …”.

That’s it. The next push or pull-request will trigger the workflow and Jest will receive:

```bash
process.env.OMDB_API_KEY === 'k3y1abc2'   # true inside the runner
```

---

### How to verify it works

1. **Push / merge** something trivial (even `README` typo) to the branch that triggers the workflow.  
2. Go to **Actions** tab → select the newest run.  
3. You should see a green check ✔️ for the **Run tests** step.  
   *If the secret is missing or misspelled you’ll get a 401/403 from OMDb and the tests fail.*

---

### Extra tips

| Need | What to do |
|------|------------|
| **Multiple repos** | Put the secret at **org level**: *Org → Settings → Secrets and variables → Actions*. |
| **Different keys per env** | Use **Environments** (e.g. `staging`, `prod`) and map secrets there. Then add `environment: staging` in the job. |
| **Rotate key** | In **Secrets** click the *pencil* icon, paste new value, save. No code change required. |
| **Local `.env` forgotten?** | CI will still pass because it gets the secret; local `npm test` will fail until you update `.env`. |

### 1. How does **Express.js** simplify server development compared to the Node.js `http` module?

| Aspect | Node.js `http` core | Express.js convenience |
|--------|--------------------|------------------------|
| **API surface** | Very low-level: you manually parse URLs, headers, query-strings, JSON bodies, form data, cookies, etc. | High-level methods like `app.get('/users/:id')`, `res.json()`, `req.body`, `req.params`, `req.cookies`—most chores handled for you. |
| **Routing** | You `if/else` or `switch` on `req.url` and `req.method`. | Declarative router (`app.get`, `app.post`, `Router()` objects) with parameter capture and middleware chains. |
| **Middleware** | You build your own composition pattern. | Built-in `.use()` pipeline; community middleware covers logging, compression, CORS, JWT auth, rate-limits, etc. |
| **Rendering / templating** | You write `res.writeHead(200, { 'Content-Type': 'text/html' });` then concat strings. | Pluggable template engines (`pug`, `ejs`, `hbs`) via `app.set('view engine', 'pug')`. |
| **Error handling** | Manually catch and map every error to HTTP status. | Central error middleware: `app.use((err, req, res, next) => res.status(500).json({error: err.message}))`. |
| **Community / docs** | You stitch libraries together yourself. | Huge ecosystem, starter generators (`express-generator`), tutorials, middleware for almost any task. |

Result: you write **significantly less boilerplate**, your code is declarative and easier to read, and you plug in battle-tested middleware instead of reinventing wheels.

**1. How Express.js makes life easier than the raw Node `http` module**

* It handles login, password hashing, and permissions.

npm keeps a list of every library your project needs (`package.json`) and the exact versions that were installed (`package-lock.json`). ASSISTANT:
**Route vs. route parameter**

* A **route** is the fixed part of the path that tells Express which handler to run, for example `/users` or `/movies`.  
* A **route parameter** is a placeholder inside that path that changes with each request, for example `:id` in `/movies/:id`. So `/movies/tt0133093` and `/movies/tt0372784` both hit the same route, and Express passes the changing part to you as `req.params.id`.

---

**What middleware does**

Middleware is any function you insert into the request-response chain before the final route handler runs. Each one gets the request, can read or change it, can add something to the response, and then calls `next()` to hand control to the next piece. Typical jobs include reading JSON bodies, adding security headers, logging, rate-limiting, and authentication checks.

---

**Custom middleware in the real world**

Imagine an internal dashboard that should be accessible only from your company’s VPN. A custom middleware could look at `req.ip`, compare it to the corporate IP range, and immediately return **403 Forbidden** if the request is coming from outside. Every protected route now needs only one line: `app.use(requireCorpVPN)`.

---

**Why order matters**

Express runs middleware in the exact order you attach them. If a logger comes after a middleware that already sent the response, nothing will get logged. If an authentication check comes after the route handler, unauthorized users will reach the handler before the check runs. Think of the chain like a stack of airport security stations: passport control must happen before you walk onto the plane, not after.

---

**Error handling in Express**

In plain JavaScript you use `try … catch` around each risky operation. Express lets you centralise that: you throw or `next(err)` inside any route or middleware, and a special error-handling middleware—defined with four arguments `(err, req, res, next)`—catches everything in one place and formats a proper HTTP response. This means you don’t have to wrap every database call with its own `try … catch`; one global handler can turn every thrown error into, say, a `500` JSON message.

---

**2 — External API call vs. database query**

*When you query your own database* you control schema, latency is inside your network, and you can shape the data any way you like.  
*When you call an external API* you go over the public internet to someone else’s server. You accept their schema, their response times, their outages, and their usage rules (keys, rate limits, pricing). In return you off-load the heavy lifting and maintenance to them.

---

**3 — Why robust error handling is vital with third-party APIs**

You can’t fix or even predict their downtime, throttling, or version changes. If a payment gateway times out, a weather API returns nonsense, or a mapping service suddenly changes its quota rules, your own app must degrade gracefully—show a friendly message, retry later, or fall back to cached data. Without defensive code you risk crashes, blank screens, or corrupted orders.

---

**4 — Risks of depending on external APIs**

* **Outages you can’t control** – the provider goes down and your key feature disappears.  
* **Breaking changes** – a version bump removes fields you rely on.  
* **Rate-limit or cost surprises** – sudden traffic spike → throttling or a huge bill.  
* **Data-privacy/legal issues** – the provider’s terms may change, or new regulations may restrict the data.  
* **Vendor lock-in** – replacing a deeply integrated API later can be expensive.

---

**5 — Handling rate limits or failures in production**

* **Caching successful responses** – store data in Redis or memory for a short time; repeated requests hit the cache, not the API.  
* **Exponential back-off retries** – on transient network errors retry 1 s, 2 s, 4 s … then give up.  
* **Graceful fallbacks** – serve stale but still useful data from yesterday, or display “Service temporarily unavailable—try again later.”  
* **Circuit-breaker pattern** – after X failures trip an “open” state that skips requests for a cooling-off period, preventing a thundering herd of retries.  
* **Monitor and alert** – log error rates, set alerts so you know when the provider is misbehaving.  
* **Budget quotas** – track remaining calls and throttle your own users before the provider slams you with 429 responses or extra fees.

Together these techniques keep your app responsive and predictable even when the outside world is not.

>  Under the hood it’s still Node, but Express removes 90 % of the boilerplate so you focus on business logic.”

**Best-practice habits for an organised Express project**

1. **Folder structure** – separate `routes/`, `controllers/`, `middlewares/`, `services/`, and `config/`.  
2. **Dotenv for secrets** – never hard-code keys; keep them in `.env` and load with `dotenv`.  
3. **Central error-handler** – one middleware that converts any thrown error into a consistent JSON response.  
4. **Async/await everywhere** – keeps flow readable; wrap with an error-catching helper.  
5. **Validation early** – use libraries like `joi` or `zod` in a dedicated middleware before the main handler runs.  
6. **Logging and request IDs** – so you can trace problems in production.  
7. **Version your API** (`/api/v1/...`) – future-proofs breaking changes.  
8. **Tests + CI** – Jest/Supertest locally, GH Actions remotely, to prevent regressions.

### A “right-sized” front-end for Movie Finder API

Below is a picture of what a **solid, portfolio-grade but still lightweight** UI would look like.  It keeps scope tight—one page and a handful of reusable components—yet shows off modern React patterns, responsive design, and good UX touches.

---

#### 1.  Single-page flow (Home)

1. **Header bar**  
   *Title + subtle GitHub link.*  Fixed at the top, uses Tailwind’s glassy backdrop so it stays readable over any poster grid.

2. **Search section**  
   *Large input field, search button, optional “Enter” key support.*  
   Below the input sit two tiny controls:  
   * “Full plot by default” switch (off by default).  
   * Page selector (previous / next arrows) once you have results.

3. **Results grid**  
   *Masonry-style card deck that auto-wraps from 2 cols on mobile to 6 cols on a wide desktop.*  Each **MovieCard** shows:
   - Poster thumbnail (fallback image if N/A)  
   - Title, year, IMDb icon with link to imdb.com/title/…  
   - **Show full plot** checkbox → when ticked the card fetches full plot via `/api/movies/:id?plot=full` and reveals it in a collapsible section.

4. **Empty state / error state**  
   *Illustration with “Try a different title”* when no movies match, and a friendly toast if the backend is down.

5. **Footer**  
   Credits OMDb and your own name; lives at the bottom of the viewport or content.

---

#### 2.  Component sketch

```
src/
├─ App.jsx                – layout shell + <Home />
├─ pages/
│   └─ Home.jsx
├─ components/
│   ├─ SearchBar.jsx      – input, button, full-plot toggle, page arrows
│   ├─ MovieCard.jsx      – poster, meta, full-plot expander
│   ├─ Toggle.jsx         – tiny reusable switch (headless-ui)
│   └─ Spinner.jsx        – centered loader
└─ hooks/
    ├─ useMovieSearch.js  – handles /api/search
    └─ useMovieDetails.js – fetch by id, cached by react-query
```

*Every component sits at 1-file ≈ 100 lines or less—easy to grok.*

---

#### 3.  State & data-fetching

* **react-query (@tanstack/query)** for request caching, loading states, and background refetch.  
  *Keys*: `['search', title, page]` and `['movie', id, fullPlot]`.

* **Context** is overkill for this size; local state (title input) and query params handle navigation.

---

#### 4.  Styling & UX polish

* **Tailwind** utility classes—no custom SCSS necessary.  
* **Dark mode** via Tailwind’s `dark:` prefix and a toggle in the header.  
* Posters get `object-cover` + subtle `hover:scale-105` animation.  
* Cards animate open/close of full plot with `@tailwindcss/typography` for nice paragraphs.  
* Mobile viewport uses `sticky` header so the search bar is always reachable.

---

#### 5.  Tooling and workflow

* **Vite** dev server with proxy: `/api` → `http://localhost:3001`.  
* **eslint + prettier** via `eslint-plugin-react` and `@tailwindcss/forms` plugin for nice inputs.  
* **Jest + @testing-library/react** for one or two smoke tests (search flow renders results, full-plot fetch triggers).  
* **Git branching**: all front-end work lives on `frontend-ui`, with PRs to merge bigger features.

---

#### 6.  Stretch toppings (post-MVP)

* **Infinite scroll** (IntersectionObserver) instead of manual page arrows.  
* **Favourites** list saved in `localStorage`.  
* **Modal** with larger poster, cast list, ratings chart.  
* **PWA** manifest + service worker so the search view is offline-cacheable.

* enough visual flair to feel polished without sinking days into design work,  
* room to add features incrementally (each as its own PR) without rewriting foundations.
