### ✅ Step 4: Add JavaScript for Interactive Behavior

---

#### 📝 Tutorial Explanation:
In this step, we make the FAQ section interactive using JavaScript. Instead of attaching individual event listeners to each question, we use **event delegation**—a powerful technique that adds one event listener to a common parent. This approach is more efficient and scalable.

We'll also handle:
- Toggling an individual answer.
- Showing or hiding all answers when Shift is held.
- Resetting (collapsing) all answers with the “Reset All” button.

---

### 📄 Create `script.js`

Create the JavaScript file:

```bash
touch script.js
```

Add this content:

```js
// script.js

const faqContainer = document.getElementById('faq-container');
const resetButton = document.getElementById('reset-button');

// Event delegation: single listener on parent container
faqContainer.addEventListener('click', (event) => {
  if (event.target.classList.contains('question')) {
    const allAnswers = faqContainer.querySelectorAll('.answer');

    if (event.shiftKey) {
      // Toggle all answers
      const anyVisible = Array.from(allAnswers).some(ans => ans.style.display === 'block');
      allAnswers.forEach(ans => {
        ans.style.display = anyVisible ? 'none' : 'block';
      });
    } else {
      // Toggle clicked answer only
      const answer = event.target.nextElementSibling;
      answer.style.display = answer.style.display === 'block' ? 'none' : 'block';
    }
  }
});

// Reset button collapses all answers
resetButton.addEventListener('click', () => {
  const allAnswers = faqContainer.querySelectorAll('.answer');
  allAnswers.forEach(ans => {
    ans.style.display = 'none';
  });
});
```

## 🔄 Automating GitHub Pages Deployment: Why & How?

### ✅ The Problem with Manual Setup:
Right now, you’ve manually enabled GitHub Pages from the UI and it publishes from the `main` branch root.

But what if:
- You want to **deploy from a `docs/` folder** or a **custom build folder** (like `dist/` or `build/`)?
- You want to **automatically deploy after each push**?
- You want to **minify, optimize, or run tests** before deployment?

This is where **GitHub Actions** shines.

---

## 🤖 What is GitHub Actions?

GitHub Actions lets you define custom workflows that trigger on events (e.g., `git push`). It automates tasks like testing, building, and **deploying**.

---

## 🚀 Use Case: Automating Deployment from a Subfolder (e.g. `docs/`)

### 📄 Why `docs/`?
GitHub Pages allows deployment from a folder named `docs/` in the root of your repo—no Actions needed. Just move your site files into `docs/`:

```bash
mkdir docs
mv index.html style.css script.js docs/
```

Then commit:
```bash
git add docs/
git commit -m "Move site files to docs/ for GitHub Pages"
git push
```

Finally:
- Go to **Settings > Pages**
- Change the source to:
  - Branch: `main`
  - Folder: `/docs`

Done! Auto-updates whenever you push to `main`.

---

## ⚙️ More Powerful Option: GitHub Actions + `gh-pages` Branch

This is useful when:
- Your site is **built from a framework** (e.g. React, Vue, Next.js)
- You **generate static files** into a `build/` or `dist/` folder
- You want to **automatically deploy only compiled output**

In this case, you'd set up an Action like:

```yaml
# .github/workflows/deploy.yml
name: Deploy to GitHub Pages

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - run: mkdir dist && cp index.html style.css script.js dist/
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist
```

This automates:
1. Pulling your latest code.
2. Preparing output (manually or via build tools).
3. Publishing it to a `gh-pages` branch used by GitHub Pages.

## 🤖 What is **GitHub Actions for CI**?

**CI** stands for **Continuous Integration**.  
In simple terms:

> 🧠 **CI is like having a robot check your code every time you push it to GitHub.**

- ✅ Run tests to make sure nothing is broken
- ✅ Check code formatting or errors (linting)

## 🧹 What is **Code Linting**?

> 🧠 Linting is like a grammar checker, but for code.

A **linter** reads your code and:
- Finds typos and syntax errors
- Warns you if you're doing something weird or risky
- Suggests improvements for cleaner, more readable code

### JavaScript Linters:
- **ESLint** is the most common one
- It checks for things like:
  - Missing semicolons
  - Unused variables
  - Bad indentation
  - `==` instead of `===` (which can be a bug)

- 🛡️ **Safety** – You catch bugs before you ship them

## 🏷️ What is a GitHub **Release**?

> Think of a **release** as an official version of your project that you can **package, tag, and describe**.

It’s like saying:

🗣️ *“Here’s version 1.0.0 of my project. It’s stable, tested, and ready for people to use or download.”*

---

### 🔹 Why Use Releases?

- ✅ Share a **stable version** with others
- ✅ Track **what changed** between versions (like changelogs)
- ✅ Download the whole project as a ZIP (from the release page)
- ✅ Mark **major updates** like "v1.0", "v2.0", "bug fix", etc.

---

### 🔧 What Does a Release Include?

1. A **tag** (like `v1.0.0`) — to identify the version
2. A **title** and **description**
3. Optionally: files, build artifacts, documentation

---

### 🛠️ In Your Case

You’ve just completed a working version of your **Interactive FAQ**. You can now:

- Tag it as `v1.0.0`
- Write: “Initial public version of the interactive FAQ project”
- Paste the **features** from your README or **changelog**

Then others (and future you!) will know: “This is a complete, working version.”

---

### 🎯 Summary

| Feature        | GitHub Release Purpose           |
|----------------|----------------------------------|
| **Tagging**    | Marks a version (`v1.0.0`)       |
| **Download**   | Zip files of that version        |
| **Tracking**   | Shows what was added/fixed       |
| **Sharing**    | Great for distribution or demos  |
