### 🌟 **TypeScript Fundamentals**

---

#### **How does TypeScript improve code quality and maintainability compared to JavaScript?**
TypeScript **checks your code while you're writing it**. It makes sure you're using the correct values and types (like string, number, etc.). This helps you **find mistakes early** instead of waiting until you run the code. This is especially useful in big projects with many files and people.

**Example:**  
If a function is supposed to take a number, TypeScript will alert you if you pass a string.

---

#### **What are the key differences between static typing in TypeScript and dynamic typing in JavaScript?**
- **JavaScript is dynamic**: you can change variable types anytime, and you won’t get errors until the code runs.
- **TypeScript is static**: you must define the type (like string, number) and the code checks for type mistakes before running.

**Example:**  
```ts
// TypeScript
let age: number = 25;
age = "twenty"; // ❌ TypeScript will catch this mistake
```

---

#### **How can enforcing type annotations in TypeScript prevent runtime errors in large-scale applications?**
By **setting types** on everything (functions, variables, etc.), TypeScript makes sure you don’t pass the wrong type accidentally. This prevents weird bugs in big apps that are hard to find later.

---

### 🛠 **Type Annotations, Interfaces, and Type Aliases**

---

#### **What are the benefits of using type annotations and type aliases in a TypeScript project?**
- **Type annotations** make your code easier to understand and catch bugs.
- **Type aliases** let you reuse complex types with simple names.

**Example:**
```ts
type UserID = string;
let id: UserID = "123abc";
```

---

#### **How do interfaces differ from type aliases, and when would you use one over the other?**
- **Interfaces** describe the shape of an object (like a blueprint).
- **Type aliases** can describe anything: a type, object, function, union, etc.

Use **interfaces** for objects, and **type aliases** when you need more flexible types.

```ts
// Interface
interface User {
  name: string;
  age: number;
}

// Type alias
type Point = { x: number; y: number };
type ID = string | number;
```

---

#### **How do interfaces and type annotations help when working with external libraries and APIs?**
They help you **know what data you're getting** and **how to use it safely**. For example, when you get data from an API, you can describe its shape so TypeScript can tell you if you’re using it wrong.

---

### 🔄 **TypeScript Generics and Advanced Types**

---

#### **What is the benefit of using generics in TypeScript, and how do they promote code reusability?**
Generics let you write **flexible and reusable code**. You don’t need to write the same function for different types. Instead, you write it once and tell it what type to use later.

**Example:**
```ts
function wrapInArray<T>(value: T): T[] {
  return [value];
}

wrapInArray(5);       // number[]
wrapInArray("hello"); // string[]
```

---

#### **How do union types and type guards improve flexibility while maintaining type safety?**
- **Union types** allow a variable to have more than one type.
- **Type guards** are checks that help TypeScript know which type is being used at a moment.

**Example:**
```ts
function print(value: string | number) {
  if (typeof value === "string") {
    console.log(value.toUpperCase()); // ✅ safe
  } else {
    console.log(value.toFixed(2)); // ✅ safe
  }
}
```

---

#### **Can you provide a real-world example where generics would be particularly useful?**
Let’s say you want to create a reusable function that returns the first item of an array — no matter if the array has strings, numbers, or objects.

```ts
function firstItem<T>(arr: T[]): T {
  return arr[0];
}

firstItem([1, 2, 3]);             // number
firstItem(["a", "b", "c"]);       // string
firstItem([{ id: 1 }, { id: 2 }]); // object
```

Using **generics** here makes your function work with any type of array — no need to rewrite it.

---

### 🔐 **Access Modifiers: public, private, protected**

---

#### **How do TypeScript's access modifiers contribute to code security and organization?**

Access modifiers control **who can see or use** a class property or method.

- **`public`**: Anyone can access it.
- **`private`**: Only the class itself can access it.
- **`protected`**: Only the class and its subclasses can access it.

**Why it matters**: It keeps your code clean and secure by **hiding internal details** you don’t want other parts of your code to mess with.

**Example:**
```ts
class User {
  public name: string;
  private password: string;

  constructor(name: string, password: string) {
    this.name = name;
    this.password = password; // Only accessible inside this class
  }
}
```

---

### 🔁 **Inheritance**

---

#### **How does inheritance reduce code duplication in a TypeScript project?**

Inheritance lets one class **reuse code from another**. If multiple classes share some functionality, you can write that once in a “base” class and extend it in “child” classes.

**Example:**
```ts
class Animal {
  move() {
    console.log("Moving...");
  }
}

class Dog extends Animal {
  bark() {
    console.log("Woof!");
  }
}

// Now Dog has both `move()` and `bark()` methods.
```

✅ You avoid rewriting `move()` in every animal type — that’s **less code, fewer bugs**.

---

### 🔁 **Polymorphism**

---

#### **Can you explain a real-world example where polymorphism would be useful in a web application?**

**Polymorphism** means that different classes can have the **same method name**, but behave differently.

**Example in a web app:**
Let’s say you have a `render()` method for UI components.

```ts
class Button {
  render() {
    console.log("Rendering a button");
  }
}

class TextBox {
  render() {
    console.log("Rendering a text box");
  }
}

function display(component: { render: () => void }) {
  component.render();
}

display(new Button());  // ✅ works
display(new TextBox()); // ✅ works
```

You can call `render()` on both, even though they behave differently. This makes code **flexible and scalable**.

---

### 🔒 **Encapsulation**

---

#### **How does encapsulation help protect sensitive data within a TypeScript class?**

**Encapsulation** means **hiding data inside a class** and controlling how it’s accessed or changed.

You use `private` or `protected` to hide the data, and `getters` or `setters` to safely access or update it.

**Example:**
```ts
class BankAccount {
  private balance: number;

  constructor(startingBalance: number) {
    this.balance = startingBalance;
  }

  getBalance(): number {
    return this.balance;
  }

  deposit(amount: number): void {
    if (amount > 0) this.balance += amount;
  }
}
```

This protects the `balance` from being changed directly, keeping your code **safe and controlled**.

ASSISTANT:
Here's a **simple, beginner-friendly explanation** of closures, higher-order functions, and their role in writing better JavaScript code:

---

### 🔒 **Closures**

---

#### **How do closures help manage variable scope in JavaScript?**

A **closure** happens when a function remembers variables from the place where it was created — even after that place is gone.

It **locks** the variables inside a function, keeping them safe from the outside. This is useful for:
- **Keeping variables private**
- **Saving data between function calls**

**Example:**
```js
function counter() {
  let count = 0;
  return function () {
    count++;
    return count;
  };
}

const myCounter = counter();
myCounter(); // 1
myCounter(); // 2
```

Here, `count` stays alive **because the inner function remembers it** — that's a closure.

---

### 🔁 **Higher-Order Functions**

---

#### **What makes higher-order functions powerful, and how can they simplify complex tasks?**

A **higher-order function** is a function that:
- Takes another function as an argument, or
- Returns another function

They help you:
- Write **less code**
- **Reuse** behavior
- **Organize** logic better

**Example:**
```js
function greet(name) {
  return "Hello, " + name;
}

function processUser(name, callback) {
  return callback(name);
}

processUser("Alice", greet); // "Hello, Alice"
```

You can swap the `callback` to do something different — that makes the code flexible.

---

### 🔄 **Callback vs. Higher-Order Function**

---

#### **What is the difference between a callback function and a higher-order function?**

- A **callback** is a function you **pass** to another function to be called later.
- A **higher-order function** is a function that **uses** or **returns** another function.

✅ A callback is **used** inside a higher-order function.

**Example:**
```js
function sayHi() {
  console.log("Hi!");
}

function runLater(callback) {
  callback(); // calling the callback
}

runLater(sayHi); // sayHi is the callback, runLater is the higher-order function
```

---

### 🧹 **Writing Cleaner Code**

---

#### **How do closures and higher-order functions contribute to cleaner, more maintainable code?**

They help you:
- **Avoid repetition** by reusing logic
- **Keep things modular** (split into small, clear pieces)
- **Protect data** (with closures)
- **Make code flexible** (swap behavior easily)

✅ This means your code is **easier to understand**, **debug**, and **change later** — especially in large apps.

ASSISTANT:
Here’s a **very beginner-friendly** explanation of error handling in modern web development and TypeScript:

---

### ❗️Why is Proper Error Handling Essential?

---

#### **Why is proper error handling essential in modern web development?**

Because **things break** — servers go down, users type the wrong thing, network fails, or bugs sneak in.

Good error handling:
- Prevents your app from **crashing**
- Gives users **helpful messages** instead of blank screens
- Helps **developers debug** faster
- Keeps your app **secure and reliable**

---

### 🔧 How `try...catch` Helps

---

#### **How does `try...catch` improve error management compared to ignoring runtime errors?**

Without `try...catch`, if something breaks in your code, it **stops everything** — and the user sees nothing or an error screen.

With `try...catch`, you can:
- **Catch the error**
- **Log it**, show a **friendly message**, or **keep going**

**Example:**
```ts
try {
  // code that might break
  let result = riskyFunction();
  console.log(result);
} catch (error) {
  console.log("Something went wrong, please try again.");
}
```

✅ You **stay in control** instead of letting your app crash.

---

### ⚙️ Custom Error Classes

---

#### **What are the advantages of creating custom error classes in TypeScript?**

A custom error class lets you:
- Add your **own message** and **error type**
- Know **exactly what went wrong**
- Handle **different errors in different ways**

**Example:**
```ts
class NotFoundError extends Error {
  constructor(message: string) {
    super(message);
    this.name = "NotFoundError";
  }
}

try {
  throw new NotFoundError("User not found");
} catch (error) {
  if (error instanceof NotFoundError) {
    console.log("Handle missing user here");
  } else {
    console.log("General error");
  }
}
```

✅ This helps you **organize your code** and **handle errors smartly**.
