## **1. The Over-fetching Problem**

In a REST API, **over-fetching** happens when the client receives more data than needed. For example, if an app only needs a user’s name and age but the API endpoint returns name, age, email, address, and phone number, it’s giving too much.

**Under-fetching** is the opposite—when an endpoint gives too little data. So the app has to make multiple API calls to gather all the information it needs.

**GraphQL fixes this** with its flexible query system. The client can ask for **exactly** what it wants, no more and no less. For example, a query can request only a user’s name and age, and the server will return just that. This helps speed up the frontend and reduces unnecessary data transfer.

---

## **2. Endpoint and Schema Philosophy**

A REST API uses **multiple endpoints**, like:
```
GET /users
GET /users/:id
GET /users/:id/posts
```
Each endpoint returns a specific kind of data. This can lead to a lot of endpoints as the app grows.

GraphQL uses **just one endpoint**, usually at `/graphql`. Instead of many URLs, it relies on a **schema** that defines all the data types and relationships. It’s like a blueprint.

For frontend developers, GraphQL’s **strongly-typed schema** is a big help. They know exactly what data is available and what they can request. It also makes tools like auto-complete and type-checking possible while writing queries, which speeds up development and reduces errors.

---

## **3. Caching and Complexity**

REST works well with caching because each endpoint returns predictable data. For example, the browser can cache `GET /posts/123` and reuse it if needed. This makes REST ideal for simple read-heavy apps.

In GraphQL, caching is harder. Since queries can be customized in so many ways, it’s tough to know when two queries are the same. Specialized tools like Apollo Client are needed to manage caching, which adds complexity.

However, **GraphQL shines when the client needs flexible and complex data**. For example, in a social media app where a page shows posts, comments, user info, and likes in one view, GraphQL can deliver it all in one request. That flexibility is worth the caching tradeoff in many real-world cases.

---

## **Conclusion & Recommendation**

Use **REST** when:
- Data structures are simple and stable
- You need good caching out of the box
- You want easier server-side setup

Use **GraphQL** when:
- The frontend requires flexible, dynamic data
- You want fewer network requests
- Developer productivity and speed are key

## Overview

This lab explores the practical differences between REST and GraphQL APIs, with a focus on long-term strategy, data fetching, caching, and development experience. The goal is to analyze the pros and cons of each approach and determine when one might be more suitable than the other.

---

## Reflection Questions

### 1. The Over-fetching Problem

In a REST API, **over-fetching** means receiving more data than needed, while **under-fetching** means not getting enough data and needing multiple requests. For example, a client may request a user’s name and get back name, email, address, and phone — more than necessary.

GraphQL solves this by letting the client ask for **only the fields it needs**. This avoids extra data, reduces bandwidth usage, and improves performance. It also avoids under-fetching because you can fetch related data in one single query.

---

### 2. Endpoint and Schema Philosophy

REST APIs use **multiple endpoints** like `/users`, `/users/:id`, and `/users/:id/posts`. Each one is fixed in structure and requires different calls to access different resources.

GraphQL uses **a single endpoint** (e.g., `/graphql`) and allows the client to send flexible queries based on a **strongly-typed schema**. This schema clearly defines what data is available and how it relates. For frontend developers, this means:
- Autocompletion when writing queries
- Clear understanding of available data
- Fewer surprises and faster development

---

### 3. Caching and Complexity

Caching in REST is easier because responses are predictable based on the endpoint and HTTP method. For example, `GET /posts/123` can be cached by the browser or a CDN.

GraphQL makes caching more complex because queries can vary widely. This makes it hard to identify what is cacheable. However, tools like Apollo Client or Relay help manage caching with normalized data.

Despite this, **GraphQL is more flexible**, especially when the frontend needs to gather complex or deeply nested data in a single request. In such cases, the trade-off is worth it.

---

## Conclusion

**Use REST when:**
- You need simple and reliable caching
- Your data model is stable and straightforward
- You want a simpler backend implementation

**Use GraphQL when:**
- Your frontend needs flexible, dynamic queries
- You want to reduce the number of API requests
- Developer speed and schema visibility are priorities

# Lab – CI/CD Pipelines Analysis

**Course:** Software Engineering Foundations  
**Lab:** CI/CD Pipelines Analysis  
**Author:** [Your Name]  
**Date:** [Insert Date]

---

## 1. The “Why”

The most important business reason to adopt a CI/CD pipeline is **speed with confidence**. When deployments are automated, the team can ship features faster without worrying about forgetting steps or breaking the app. It reduces human error, shortens the time from writing code to releasing it, and gives the team more time to focus on building instead of manually managing deployments. For the product, this means users get updates more quickly and reliably, improving their experience and trust.

---

## 2. The “How”

**Continuous Integration (CI)** means automatically testing and merging code changes into the shared codebase. It ensures that the code from different developers works well together. CI ends when the code is built and tested successfully.

**Continuous Deployment (CD)** takes over after CI. It automatically deploys the tested code to production or staging environments. While CI checks if the code is good, CD pushes that code out to real users — without needing a person to step in.

In simple terms:  
- CI = merge, build, and test the code  
- CD = release the code to users

---

## 3. Tool Comparison

For a small, budget-conscious startup already using GitHub, **GitHub Actions** is the best choice.

**Reasons:**
- **Native GitHub Integration**: Since the team already uses GitHub, setup is easier and workflows are directly tied to GitHub events like pushes and pull requests.
- **Free Tier**: GitHub Actions offers generous free minutes for public and private repos, which is ideal for startups trying to save costs.

It’s also simpler to learn and manage compared to Jenkins, which requires self-hosting and more setup.

---

## 4. Flexibility vs. Simplicity

A scenario where **Jenkins** is better than GitHub Actions would be a **large enterprise** with a complex infrastructure. For example, if the team needs to deploy to multiple cloud providers (AWS, Azure, GCP) with custom scripts, manual approval gates, and advanced plugin needs, Jenkins provides the flexibility to support this.

Even though Jenkins requires more maintenance, it gives full control and customization over the entire CI/CD process — perfect for companies with strict deployment rules or legacy systems.

---

## 5. Personal Connection

In a past project, I worked on a web app where, after pushing to GitHub, I had to manually:
1. Pull code to the server via SSH
2. Run `npm install`
3. Rebuild with `npm run build`
4. Restart the Node.js server

This was slow and easy to mess up. A CI/CD pipeline could have handled all of it automatically whenever I pushed to the `main` branch. This would’ve saved time and reduced the chance of forgetting steps — like the time I forgot to restart the server and thought my code was broken!
