# Gaps / risks to resolve for a successful MVP
(These are common gaps I noticed from the project context and the concept — addressing them will help the product actually work for street vendors.)

1. **Payments & settlement** — need an easy on-ramp: card, mobile wallet, cash-on-pickup, and a simple reconciliation/merchant payout flow.  
2. **Offline / intermittent connectivity** — vendors on the street often have poor connectivity; need caching, local queueing, or an SMS fallback.  
3. **Order routing & notifications** — real-time order push to vendor dashboard + audible/paged alerts (and SMS fallback).  
4. **Inventory / availability** — simple “sold out” flags per item + quick menu edits from vendor mobile.  
5. **Fees & legal** — commission model, receipts, refunds, terms of service and local food vendor rules.  
6. **Onboarding & training** — extremely simple setup flow for non-technical vendors (photo menu, pricing, QR generation).  
7. **Analytics & dispute resolution** — basic reports + support workflow for order problems.  
8. **UX consistency** — standardize style (fonts, spacing, button states) across vendor/customer flows (you asked about UX consistency previously).

# Prioritized MVP checklist (recommended order)
1. Core ordering flow end-to-end: QR scan → menu → add to cart → place order → vendor receives order.  
2. Vendor dashboard: accept/reject order, mark complete, simple order queue.  
3. Payment options: start with **cash on pickup** + one digital gateway (Stripe/PayPal) for cards.  
4. Notifications: in-app + SMS fallback for new orders.  
5. Menu editor for vendors: add/edit item, toggle availability, upload photo.  
6. Generate & print/share QR for each vendor/location.  
7. Basic reporting: daily orders, revenue, AOV.  
8. Onboarding flow & one-page cheat sheet for vendors.  
9. UX polish and responsive mobile-first QA.  
10. Pilot plan & legal (merchant agreement + local compliance).

# Quick technical tasks (dev checklist)
- Wire up real-time DB listeners for orders (Firebase Realtime / Firestore onSnapshot).  
- Implement order states: `pending → accepted → preparing → ready → completed`.  
- Add push notifications (Firebase Cloud Messaging) + SMS gateway (Twilio) fallback.  
- Integrate payment gateway sandbox (Stripe Connect if you’ll pay out vendors).  
- Implement offline queueing / retry for vendors (localStorage + background sync pattern).  
- Create endpoints or cloud functions to handle payouts, receipts, and refunds.  
- Add analytics events (order_placed, order_accepted, order_completed, vendor_onboarded).

# MVP success metrics / KPIs to track
- Number of active vendors onboarded (goal for pilot: 10–30).  
- Conversion rate (QR scans → orders).  
- Average order value (AOV).  
- Orders per vendor per day.  
- Time vendor takes to accept & complete an order.  
- Customer repeat rate (7-day retention).  
- Payment success / failure rate.

# Suggested pilot plan (fast, low-risk)
- Pick a neighborhood or market; onboard 10 vendors with in-person help.  
- Offer zero commission for first 2–4 weeks to encourage adoption.  
- Collect structured feedback (3 questions) after each order for first week.  
- Iterate UI for vendor pain points (notifications, connectivity).

# UX / product tips (based on your preferences)
- Keep vendor UI minimal: two large buttons (“Accept”, “Decline”), big timer, order summary with photos.  
- Use progressive disclosure: advanced features (promos, inventory) hidden behind a “Manage” tab.  
- Provide a printable one-page QR + instructions for each vendor.
