---

## 1. Decide the goal of the refactor

Pick what you want out of this:

- ✅ Easier to read/maintain (naming, structure)
- ✅ Separate logic from HTML (MVC-ish structure)
- ✅ Prepare to move to a framework later (Laravel, Symfony, etc.)
- ✅ Improve security (SQL injection, XSS, CSRF, secret handling)
- ✅ Make it testable (unit tests, at least for business logic)

Knowing the goal defines how deep we go.

---

## 2. Get it running as-is

Before touching anything:

1. Unzip into a working folder.
2. Make sure you can run it on:
   - Local server (XAMPP/MAMP/Laragon/Docker)  
   - Confirm PHP version (e.g. 7.4, 8.1, 8.2).
3. Open in VS Code or your editor of choice.

If it doesn’t run cleanly, we fix *runtime errors first*, then refactor.

---

## 3. Understand the current structure

Look at:

- Entry point(s): `index.php`, `home.php`, etc.
- How pages are linked: direct `.php` links or routing system?
- Where DB logic lives: inline queries in pages, or a `db.php` file?
- Any libraries/composer? (Look for `composer.json` or `vendor/`)

Typical mess we’ll likely see:

- PHP + HTML all mixed in the same file
- Raw SQL with `mysqli_*` or `mysql_*`
- Big “god files” with lots of responsibilities
- Globals everywhere (`$_POST`, `$_GET`, `$_SESSION` inside views)

---

## 4. Define a target structure (lightweight MVC)

Even without a full framework, we can aim for something like:

```text
project/
  public/
    index.php       # front controller
    assets/         # css, js, images
  app/
    Controllers/
    Models/
    Views/
    Helpers/
  config/
    config.php      # db settings, env vars
  vendor/           # if using composer
```

Principles:

- **Controllers**: get request, call models, load views.
- **Models**: DB and business logic.
- **Views**: mostly HTML with minimal PHP echoing variables.

---

## 5. Step-by-step refactor strategy

We’d do this gradually, not all at once:

1. **Extract configuration**
   - Move DB credentials, API keys, etc. into `config/config.php` or `.env`.
   - Make the rest of the app read from there.

2. **Introduce Composer & autoloading (if not present)**
   - Create a `composer.json`.
   - Use PSR-4 autoloading for `App\...` namespaces.
   - Remove manual `require`/`include` chains where possible.

3. **Separate DB logic**
   - Find all raw queries and centralize them into Model classes or a `Database` helper.
   - Replace inline SQL in views/pages with calls to these models.

4. **Split giant pages**
   - Take any big `.php` page with HTML + logic:
     - Move logic into a controller or helper.
     - Pass data into a view file using variables.
   - Leave only something like:

     ```php
     <?php include 'header.php'; ?>
     <h1><?= htmlspecialchars($title) ?></h1>
     <?php foreach ($items as $item): ?>
       ...
     <?php endforeach; ?>
     <?php include 'footer.php'; ?>
     ```

5. **Routing**
   - If the project is `page1.php`, `page2.php` style, we can:
     - Introduce a single `public/index.php` and route via `?route=...` as a first step, or
     - Use something like `index.php?page=home` and map that to controllers.
   - Later: swap to a real router (e.g. a tiny router package or a micro-framework).

6. **Improve security**
   - Replace string-concatenated SQL with prepared statements.
   - Use `htmlspecialchars` on all user-visible data.
   - Check form handling and session management.

7. **Add basic tests**
   - Even just a few PHPUnit tests around the most critical logic (price calculations, validations, etc.) makes refactors safer.
