### Why this happens

`npx` looks inside each package for a **`bin`** field.  
If the install was interrupted or an old cache entry is broken, `npx` can’t locate the executable and emits that error. Forcing a reinstall or using `npm exec` (which consults your project’s `node_modules/.bin` first) resolves it 99 % of the time.

### Short answer  
No—you don’t *have* to touch **App.css** at all. With Tailwind CSS we style almost everything with **utility classes (e.g. `bg-charcoal text-gold`)**, and the colour tokens themselves live in **`tailwind.config.ts`**. The fact that your page is still white / black means the custom palette isn’t being picked up yet, not that App.css needs edits.

Below is a quick checklist to get the palette loading, then we can safely delete `App.css` (or keep it empty for one-off styles later).

---

## 1  Confirm the Tailwind config is in the right place & name

Tailwind (and the Vite plugin) look for one of these filenames in the **project root**:

```
tailwind.config.js     ✅ always works
tailwind.config.cjs    ✅
tailwind.config.mjs    ✅
tailwind.config.ts     ✅ (supported, but must export `default`)
```

If yours is `tailwind.config.ts` make sure it’s sitting next to `vite.config.ts`, not inside `/src`.

```bash
seven-teas/
├─ tailwind.config.ts   <-- here
├─ vite.config.ts
└─ src/
```

## 3 Verify the config file name & location

```
seven-teas/
├─ tailwind.config.js     <-- CommonJS (`module.exports = { ... }`)
├─ vite.config.ts
├─ postcss.config.js      <-- optional; fine if it exists
└─ src/
   └─ index.css
```

Tailwind’s loader **will ignore** `tailwind.config.ts` unless your Node project is set to `"type": "module"` in `package.json`. Stick with `tailwind.config.js` (CommonJS) for now.
