```ts
// 1. Closures
function createMultiplier(multiplier: number): (value: number) => number {
    return function (value: number): number {
        return value * multiplier;
    };

    return function (price: number): number {
        return price - price * rate;
    };
}

console.log(tenPercentDiscount(100)); // Output: 90

// 3. Callback Example
function fetchUser(username: string, callback: (user: { username: string, id: number }) => void): void {
    setTimeout(() => {
        const user = {
            username,
            id: Math.floor(Math.random() * 1000),
        };
        callback(user);
    }, 1000);
}

fetchUser("Alice", (user) => console.log("Fetched User:", user));
```

### 🧠 Critical Thinking

**1. How do closures help manage variables in asynchronous operations?**  
Closures capture the surrounding lexical scope, allowing variables to persist beyond their initial execution context. This is especially useful in asynchronous code, where the original stack frame may be gone by the time a delayed callback is executed.

**Example:**
```js
function asyncLogger(message) {
  setTimeout(() => {
    console.log("Logged:", message); // message is preserved by the closure
  }, 1000);
}
```

**2. Why are higher-order functions valuable for building reusable code?**  
They allow abstraction over actions, not just values. By taking or returning functions, they support modularity, DRY principles, and dynamic behavior injection (e.g., passing different strategies, operations, or policies).

**3. In what situations would you choose callbacks over promises or async/await?**  
- When integrating with older APIs or libraries that rely on the callback pattern.
- In low-level event-driven environments (e.g., Node.js streams, hardware I/O).
- When memory and performance are critical, and callback style avoids wrapping extra promise objects.

The HTML you posted appears to be fine syntactically, so if you're getting an **"error"** when running this in the browser, here are the most likely causes and how to fix each:

---

### ✅ 1. **Check TypeScript Was Compiled**
Make sure the `.ts` was compiled into `.js`:
```bash
npx tsc
```

Then verify that:
```
./dist/main.js
```
**exists**. If not, there’s likely a misconfiguration in `tsconfig.json`.

---

### ✅ 2. **Check `tsconfig.json` Paths**
Ensure:
- `rootDir` is set to `"./src"`
- `outDir` is set to `"./dist"`

Minimal example:
```json
{
  "compilerOptions": {
    "target": "ES6",
    "module": "ES6",
    "rootDir": "./src",
    "outDir": "./dist",
    "strict": true
  }
}
```

---

### ✅ 3. **Browser Won’t Show Output**
If your browser shows a blank page or just HTML:
- **Open DevTools** (F12 → Console tab)
- You may see:
  - `Failed to load module script: Expected a JavaScript module script but the server responded with a MIME type of "text/html"` → This means `main.js` was **not found**, and your browser loaded a fallback 404 HTML page **instead of** the script.

✅ Fix: Confirm `./dist/main.js` exists and is **not empty**.

---

### ✅ 4. **Live Server Needed for Module Imports**
If you’re running from local file (`file://`) and using:
```html
<script type="module" src="./dist/main.js"></script>
```
then the browser may block it.

✅ Fix:
- Use **Live Server** extension in VS Code.
- Or serve with a local server:
  ```bash
  npx serve .
  ```

---

### ✅ 5. **Compile Errors in TypeScript**
If `main.ts` has a TypeScript error, `main.js` won’t be generated.

✅ Fix:
```bash
npx tsc --noEmitOnError false
```
to see the error and allow emit for debugging.
