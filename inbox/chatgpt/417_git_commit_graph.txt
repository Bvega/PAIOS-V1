USER:
ich command displays a graphical representation of the commit history?

--------------------------------------------------------------------------------

ASSISTANT:
Use Git’s built‑in graph view. For example, in your repo run:

```bash
git log --graph --oneline --decorate --all
```

This will show an ASCII‑art commit graph in your terminal. If you want a GUI, you can use:

```bash
gitk --all
```


(or equivalently `git diff --cached`) to see exactly what’s in the index and ready to be committed.

USER:
What is the purpose of the git remote command?

--------------------------------------------------------------------------------

ASSISTANT:
The `git remote` command lets you view and manage the “remotes” (other repositories) your local repo tracks.  

- `git remote`  
  Lists the short names of all remotes (e.g. `origin`, `upstream`).  
- `git remote -v`  
  Shows each remote’s name plus its fetch/push URLs.  
- `git remote add <name> <url>`  
  Adds a new remote under the given name.  
- `git remote remove <name>`  
  Deletes that remote.  
- `git remote rename <old> <new>`  
  Renames a remote.  

You use it to configure where you fetch from and push to.

--------------------------------------------------------------------------------

USER:
Which command is safest for undoing a commit that has already been pushed to a shared repository?

--------------------------------------------------------------------------------

ASSISTANT:
Use `git revert` to undo it safely. For example, to revert the last pushed commit:

```bash
git revert HEAD
```

This creates a new commit that undoes the changes, preserving history and avoiding force‑pushes.

Which of the following is a best practice when writing commit messages?

--------------------------------------------------------------------------------

ASSISTANT:
Use a concise, 50‑character‑or‑less subject line written in the imperative, present tense (e.g. “Add user authentication”), separated from a more detailed body (wrapped at ~72 chars) by a blank line.

--------------------------------------------------------------------------------

--------------------------------------------------------------------------------

ASSISTANT:
- Write the **subject line** in the **imperative, present tense** (e.g. “Fix typo in README,” not “Fixed…” or “Fixes…”).  
- Keep the subject to **50 characters or fewer**, and capitalize it without a trailing period.  
- Separate the subject from the body with a **blank line**.  
- If you need more detail, use the **body** to explain the “what” and “why,” wrapping lines at **72 characters**.  
- Reference any relevant issues or tickets (e.g. “Refs #123”) in the body.
