## Asynchronous Programming with Promises

1. **Difference between synchronous and asynchronous code**  
   - **Synchronous code** executes line by line; each operation must complete before the next starts. If one step blocks (e.g., heavy computation or I/O), everything else waits.  
   - **Asynchronous code** allows operations to start and then move on without waiting for them to finish—using callbacks, Promises, or async/await. This non-blocking behavior is crucial in web development to keep the UI responsive (e.g., handling user input, animations) while performing network requests or file I/O in the background.

2. **How Promises help manage asynchronous operations**  
   - A **Promise** represents a value that may be available now, later, or never. It solves the “callback hell” problem by allowing you to chain `.then()` calls for sequencing instead of nesting callbacks.  
   - Promises provide a clear structure for success (`resolve`) and failure (`reject`), making asynchronous code more readable and maintainable.

3. **Importance of handling rejected Promises with `.catch()`**  
   - If a Promise is rejected and you don’t call `.catch()`, the error goes unhandled, potentially causing silent failures or unanticipated application states.  
   - `.catch()` ensures you can respond to errors (e.g., show an error message, retry logic) and prevents unhandled rejection warnings in the console.

## Async/Await and the Event Loop

4. **How async/await simplifies Promises**  
   - `async` functions let you write asynchronous code that looks synchronous. Inside an `async` function, you can `await` a Promise, pausing execution until it resolves or rejects.  
   - This flattens chains of `.then()` calls and makes try/catch error handling natural.

5. **Role of the JavaScript event loop**  
   - The **event loop** is the engine that coordinates the call stack (synchronous code) and the task queue (asynchronous callbacks, Promise callbacks, timers, I/O).  
   - When the call stack is empty, the event loop picks the next task from the queue and pushes its callback onto the stack, ensuring non-blocking concurrency.

6. **Error handling with async/await**  
   ```javascript
   async function fetchData() {
     try {
       let response = await fetch('/api/data');
       if (!response.ok) throw new Error('Network response was not ok');
       let data = await response.json();
       return data;
     } catch (err) {
       console.error('Fetch failed:', err);
       // handle error, e.g., return fallback value
     }
   }
   ```
   - Wrapping `await` calls in `try/catch` catches both Promise rejections and thrown errors for cleaner, linear control flow.

## Working with the Fetch API

7. **Purpose of the Fetch API**  
   - Fetch provides a modern, Promise-based interface for HTTP requests, replacing older XHR (XMLHttpRequest).  
   - It simplifies making requests (`fetch(url, options)`), handling JSON payloads (`response.json()`), and chaining operations.

8. **Common pitfalls & avoidance**  
   - **Not checking `response.ok`**: A 404 or 500 won’t reject the Promise—you must manually check `response.ok` and throw if false.  
   - **Forgetting to parse**: `fetch` resolves with a `Response` object; you still need to parse it (`.json()`, `.text()`, etc.).  
   - **CORS issues**: Ensure the server provides the appropriate Cross-Origin headers.

9. **Handling successful and unsuccessful responses**  
   ```javascript
   fetch('/api/data')
     .then(response => {
       if (!response.ok) {
         return response.text().then(msg => { throw new Error(msg); });
       }
       return response.json();
     })
     .then(data => {
       // handle data
     })
     .catch(err => {
       // handle network error or non-2xx status
       console.error('Request failed:', err);
     });
   ```
   - Always check `response.ok`, parse error messages if available, and use `.catch()` to handle both network failures and HTTP errors.
