# Flaw analysis: the 7-step outline (for coding agents)

The list is a decent skeleton but **too coarse for real agentic coding**. Biggest gaps:

1) **Vague problem framing**  
   - No product/UX spec, success criteria, or non-goals.  
   - For coding agents, you need explicit repository scope, languages, frameworks, code-ownership rules, and allowed operations (read-only vs write/PR).

2) **Model-centric bias**  
   - Jumps to “collect data → train model.” Most agentic coding systems are **tool-orchestrated**; you likely won’t train your own model at first. Focus should be on **tooling + control flow** (planner/worker/reviewer loops) and **evaluation**, not training.

3) **No task decomposition or planning**  
   - Missing an explicit **planner** that breaks a user goal into verifiable subtasks (e.g., write failing test → implement change → run tests → generate diff → propose PR).

4) **Missing execution sandbox & safety**  
   - Coding agents must run code/tests safely. Absent: containerization, resource limits, network policy, file write policy, SBOM, dependency pinning.

5) **No memory/state design**  
   - Agents need **short-term scratchpads** and **long-term project memory** (repo index, architectural map, API docs). Without it: repetition, context loss, and brittle behavior.

6) **Insufficient evaluation**  
   - “Test and improve” is underspecified. You need **gold tasks**, **unit/integration tests**, offline/online metrics, **hallucination/error taxonomies**, and **regression gates**.

7) **No guardrails or governance**  
   - Missing timeouts, retries, budget caps, circuit breakers, escalation to human, approval steps before writing/committing code, and **audit logs**.

8) **Integration is underspecified**  
   - How does the agent talk to: Git, CI, package manager, issue tracker, documentation search, code search? Tool specs are essential.

9) **Observability gap**  
   - Absent tracing, structured logs, event timelines, per-tool latency/cost, and failure analytics. Without this you can’t iterate.

10) **Security & privacy**  
   - No policy for secrets handling, least-privilege credentials, repo access scopes, or red-teaming prompt injection in docs/code.

11) **Versioning & rollout**  
   - No model/prompt versioning, feature flags, A/B canaries, or rollback plan.

12) **Human factors**  
   - Lacks human-in-the-loop checkpoints (PR review gates, dependency updates). No UX for approvals or diff visualization.

# Recommendations for an implementation (PENDING APPROVAL)

## A) Scope & success criteria
- **Problem class**: “Agent that proposes code changes via PRs for a defined repo set.”  
- **Allowed actions**: read repo, write diffs to a branch, open PR with justification; **no direct commits to main**.  
- **Success**: % tasks completed, test pass rate, review acceptance rate, rework count, cost/latency budget adherence.

## B) System architecture (agent graph)
- **Planner**: decomposes request → plan with measurable milestones (tests/diffs).  
- **Retriever**: repo index + docs/issue search (semantic + symbol-aware code search).  
- **Implementer**: edits code with constraints; runs local tests.  
- **Reviewer**: static analysis, style/lint, security scan, generates review notes.  
- **PR Agent**: compiles diff, rationale, risk notes, and test evidence; opens PR and requests human review.  
- **Supervisor**: budgets, timeouts, retries, circuit breakers, escalation.

**State/memory**  
- **Short-term**: per-task scratchpad (plan, assumptions, artifacts).  
- **Long-term**: project index (vector + symbols), decision log, failure patterns, approved patterns.

## C) Tooling (minimum viable tool spec)
- **Repo & CI**: Git (read/write branch), PR API, CI status read.  
- **Execution sandbox**: container with CPU/RAM caps, no outbound network by default; allow curated mirrors if needed.  
- **Test runner**: language-specific (npm/yarn/pnpm + jest/vitest; pytest; mvn/gradle, etc.).  
- **Static analysis**: linters, formatters, SAST (e.g., ESLint/Prettier, Bandit, Semgrep).  
- **Package mgr**: lockfile enforcement, vuln scan.  
- **Search**: code symbol index + embeddings; doc search with prompt-injection filters.  
- **Diff/patch**: unified diff creation, patch apply/rollback.  
- **Issue tracker**: read issues/acceptance criteria, post status updates.

## D) Control policies
- **Write policy**: only to a feature branch; max files changed; max LOC per patch; require failing test first when feasible.  
- **Budget policy**: cap tokens/time/tool-invocations per task; kill-switch.  
- **Risk policy**: no secrets; no dependency upgrades beyond patch unless ticket permits; no infra changes without gate.

## E) Evaluation & QA
- **Gold task suite**: 25–100 representative tasks (bug fixes, small features, refactors).  
- **Metrics**: task success, test pass@k, static analysis errors, revert rate, PR acceptance rate, human edit distance, cost/latency.  
- **Taxonomy**: label failures (wrong file, misunderstanding API, flaky tests, env mismatch, hallucinated symbol).  
- **Regression gates**: changes to prompts/models/parameters must equal or beat baseline.

## F) Safety, security, governance
- **Sandbox**: seccomp/cgroups limits; read-only filesystem except working dir; redact env; secrets vault (read-only tokens).  
- **Prompt defense**: strip/neutralize instructions from repo docs; mark untrusted content; dual-model confirmation for dangerous actions.  
- **Audit**: structured logs, trace IDs per task, full artifact bundle (plan, diffs, test output, prompts).  
- **Human gates**: mandatory human approval for PRs; escalation path on low confidence.  
- **Versioning**: semantic versions for prompts/graphs; signed releases; rollback.

## G) Rollout plan
1. **Pilot** on a single repo (read-only dry-run → shadow mode → PR-only mode).  
2. **Reviews**: 10–20 PRs reviewed by maintainers; collect acceptance/rework.  
3. **Harden** guardrails from post-mortems; expand toolchain.  
4. **Scale** to more repos; enable canary % for auto-suggested PRs.

## H) Implementation phases (2–3 weeks each)
- **Phase 0**: ADR-001 architecture + risk register; define gold tasks; select models.  
- **Phase 1**: Repo index/search + planner/implementer loop; dry-run planner outputs only.  
- **Phase 2**: Sandbox + test runner + diff generator; PRs behind human gate.  
- **Phase 3**: Reviewer agent (lint/SAST) + telemetry + dashboards.  
- **Phase 4**: Eval harness/regression; policy tuning; canary rollout.

## I) Minimal configuration decisions to lock before build
- Target language(s)/framework(s) per repo.  
- CI runner image & base container.  
- Tool list (linter/test frameworks).  
- Approval UX (who reviews PRs, SLA).  
- Budget caps and maximum patch size.
